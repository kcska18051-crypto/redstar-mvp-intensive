# Программа: от идеи до работающего MVP

## Соглашения курса (прочитай один раз)

На курсе ты работаешь с **AI-агентом** в редакторе кода. Это может быть **Claude Code** (от Anthropic) или **Codex** (от OpenAI) — команды и логика одинаковые. Где есть различия, отмечаем так: **🔵 Claude Code** / **🟢 Codex**.

**Про модели.** У каждого агента есть быстрые и мощные модели. В заданиях пишем нейтрально:
- **Модель: быстрая** — для простых задач (🔵 Claude: Sonnet / 🟢 Codex: быстрая модель).
- **Модель: мощная** — для сложных задач с рассуждением (🔵 Claude: Opus / 🟢 Codex: мощная reasoning-модель, reasoning high).

Конкретные названия моделей у обоих агентов меняются часто — выбирай по принципу «быстрая / мощная», а не по имени.

**Автономный режим.** Чтобы агент не спрашивал разрешение на каждое действие — включи автономный режим. Для учебного проекта это нормально.
- **🔵 Claude Code:** включи автономный режим — режимы доступа переключаются клавишей **Shift+Tab** (или в настройках агента). Выбери режим, в котором агент сам правит файлы и выполняет команды без подтверждения каждого шага.
- **🟢 Codex:** режим доступа выбираешь командой **`/permissions`**. Дефолтный режим **Auto** правит файлы и запускает команды только ВНУТРИ рабочей папки — а для заданий 11 и 14 этого мало: там SSH на сервер, установка PostgreSQL и деплой, то есть команды ВНЕ папки проекта. В режиме Auto Codex будет переспрашивать (или откажется) на каждой серверной команде, и покажется, что «он сломался». Поэтому на серверных шагах включай **Full Access**.

> На курсе автономный режим экономит время. Для реальных проектов потом используй более осторожные режимы — и никогда не отключай проверки безопасности на боевых серверах.

## Что мы делаем и в каком порядке

| Фаза | Задания | Что делаем | Результат |
|------|---------|-----------|-----------|
| 1. Знакомство с AI-агентом | 1-4 | Учимся работать с файлами через AI | Тренировочные файлы в `1-2-MVP/results/` |
| 2. Продуктовые требования | 5-6 | Описываем продукт через PRD-интервью | `1-2-MVP/results/PRD.md` |
| 3. Лендинг (маркетинг) | 7-10 | Делаем рекламную страницу для привлечения пользователей | `1-2-MVP/results/landing.html` |
| 4. MVP (сам продукт) | 11-15 | Пишем и деплоим работающее приложение | Код в `product-mvp/`, приложение на сервере |

> **Лендинг и MVP — это два разных проекта.**
> Лендинг — одностраничный сайт, который РАССКАЗЫВАЕТ о продукте (маркетинг).
> MVP — работающее приложение, которым пользователь ПОЛЬЗУЕТСЯ (сам продукт).
> Мы делаем их по очереди, но это разные вещи с разным результатом.

# Воркшоп 2: Создание и деплой MVP

## Фаза 4: MVP — работающий прототип продукта

> **Что это:** MVP (Minimum Viable Product) — это работающее приложение, которым пользователь реально ПОЛЬЗУЕТСЯ.
> В отличие от лендинга, MVP не просто рассказывает о продукте — он ДЕЛАЕТ то, что обещает.
>
> **MVP и лендинг — это разные проекты.**
> Лендинг (фаза 3) — статичная рекламная страница для привлечения пользователей.
> MVP (эта фаза) — работающее приложение с логикой, данными и интерфейсом.
>
> Мы берём продуктовые требования из `1-2-MVP/results/PRD.md` и пишем настоящий код приложения.

**Результат фазы:**
- Работающее приложение задеплоено на сервер и доступно по ссылке
- Приложение реализует ключевые фичи из PRD.md
- Продуктом можно пользоваться — основной сценарий работает от начала до конца

---

## Задание 11: Пишем код MVP (40-60 минут)

**Модель:** мощная

**Зачем вам это уметь:** Вы превращаете документ (PRD) в работающий код. Это главный навык курса — умение пройти путь от идеи до прототипа за один вечер, а не за месяц.

> Мы начинаем писать ПРОДУКТ — рабочее приложение. Это НЕ лендинг.
> Лендинг уже готов в `1-2-MVP/results/landing.html`, MVP живёт отдельно — в `product-mvp/`.

**Стек технологий:**

| Компонент | Технология | Зачем |
|-----------|------------|-------|
| Фреймворк | **Next.js** (App Router) | Фронтенд + API в одном проекте |
| Стили | **Tailwind CSS** | Быстрая вёрстка без написания CSS |
| База данных | **PostgreSQL** (на учебном VPS) | Хранение данных продукта |
| Деплой | **Учебный VPS** | Приложение доступно по IP |

> **Про базу данных:** Агент сам подключится к вашему учебному VPS, установит и настроит PostgreSQL, создаст таблицы и заполнит тестовыми данными.
>
> **Не пугайтесь стека** — агент сам создаст проект, установит зависимости и настроит всё. Вам не нужно знать эти технологии заранее.

**MVP — это отдельный проект.** Он живёт в папке `1-2-MVP/product-mvp/` и НЕ должен лежать в `1-2-MVP/results/` (там — лендинг и PRD). Для сборки есть пошаговая инструкция (стек, база данных, дизайн, деплой) — файл `1-2-MVP/prompts/mvp.md`.

**Введите агенту:**
```
Разработай работающее приложение (MVP) в папке 1-2-MVP/product-mvp/ — это сам продукт, а НЕ лендинг (лендинг уже готов в 1-2-MVP/results/).
Опирайся на два файла:
- продуктовые требования: 1-2-MVP/results/PRD.md (что за продукт и какие функции);
- инструкция по сборке: 1-2-MVP/prompts/mvp.md (стек, база данных, дизайн, деплой).
Весь код создавай прямо в product-mvp/, без вложенных подпапок.
```
